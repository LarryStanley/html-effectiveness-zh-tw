INC-2025-0412

任務同步出現異常大量 502 錯誤

SEV-2 已解決 持續時間 47 分鐘 偵測時間 4 月 12 日 · 14:07 負責人 Devon Park
一句話重點（TL;DR）

一次 config 部署把 sync-worker 層級的資料庫連線池上限從 64 降到 8，結果在平常的下午流量尖峰就吃不消了，連線被耗盡。在這 47 分鐘內，大約 21% 的同步 API 請求都收到 502 錯誤。我們的處置是直接還原 config，然後把整個 worker 機群重啟一遍；沒有發生任何資料遺失。

事件時間軸

14:02
Config 變更 cfg-9a12 透過標準的部署流程發佈到生產環境。
14:06
影響開始。Sync worker 在取得連線池時開始卡住排隊；p95 延遲飆破 4 秒，負載平衡器開始回傳 502 錯誤。
14:07
警報觸發：sync_5xx_rate > 2% 持續 60 秒。值班人員（Devon）確認接手。
14:18
一開始猜是 API 服務的部署出包，於是把前兩次的應用程式部署都還原了，但完全沒用。
14:31
Mira 加入幫忙，看到 worker dashboard 裡面連線池等待的數值已經滿了。調查方向於是轉向基礎設施的 config，而不是應用程式碼。
14:44
狀況緩解。已還原 cfg-9a12，並重啟 worker 機群。5xx 錯誤率在三分鐘內就降到 0.2% 以下。
14:49
監控恢復正常持續 5 分鐘。宣告事故解決，並更新狀態頁。

根本原因

PR #4888 把連線池上限改成可以按 worker 層級各自設定。新增的 sync-worker 這個 key 本來應該要繼承全域預設值（64），但在本機測試時被寫死成 8，然後就這樣 commit 出去了。而 config 的 linter 只檢查了型別，沒檢查數值大小合不合理，所以這個變更就在 CI 通過了。

因為 config 部署跟程式碼部署是走兩條獨立的 pipeline，值班人員一開始直覺反應去還原最近的應用程式部署，結果當然沒效，這下子白白耗掉了大約 13 分鐘的排查時間。

infra/config/workers.yaml
pool:
global_max_connections: 64
tiers:
- sync-worker: { max_connections: 64 }
+ sync-worker: { max_connections: 8 } # debug value, do not ship
webhook-worker: { max_connections: 32 }

影響範圍

失敗的請求（502） 約 41,200
最高錯誤率 21.4%
受影響使用者 約 2,300 個工作區
資料遺失 無 — 客戶端都會重試
SLA 違約 無（在每月容許額度內）

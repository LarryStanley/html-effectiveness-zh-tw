04

關鍵程式碼

有兩個地方最容易出錯：一是 migration（軟刪除與閱讀進度），二是樂觀寫入的 mutation（暫時 ID 的對齊）。

packages/db/migrations/0042_comments.sql
create table comments (
  id          uuid primary key default gen_random_uuid(),
  task_id     uuid not null references tasks(id),
  parent_id   uuid references comments(id),   -- 只允許一層巢狀，
                                                -- 由 API 強制限制
  author_id   uuid not null references users(id),
  body        text not null,
  created_at  timestamptz not null default now(),
  deleted_at  timestamptz                       -- 軟刪除欄位
);

create table comment_reads (
  task_id    uuid not null references tasks(id),
  user_id    uuid not null references users(id),
  read_up_to timestamptz not null,
  primary key (task_id, user_id)
);

create index comments_task_created
  on comments (task_id, created_at);
apps/web/hooks/useAddComment.ts
export function useAddComment(taskId: string) {
  const qc = useQueryClient();
  return trpc.comments.create.useMutation({
    onMutate: async (input) => {
      const temp = { ...input, id: `temp-${nanoid()}`,
                     createdAt: new Date(), pending: true };
      qc.setQueryData(key(taskId), (prev) =>
        [...(prev ?? []), temp]);
      return { tempId: temp.id };
    },
    onSuccess: (row, _v, ctx) => {
      // 把暫時 ID 對應成真實 ID，
      // 這樣即時推送進來時才不會重複顯示
      qc.setQueryData(key(taskId), (prev) =>
        prev.map((c) => c.id === ctx.tempId ? row : c));
    },
    onError: (_e, _v, ctx) => {
      qc.setQueryData(key(taskId), (prev) =>
        prev.filter((c) => c.id !== ctx.tempId));
    },
  });
}

風險與緩解措施

風險項目
嚴重度
緩解措施
即時重複問題：WebSocket 推送跟 HTTP 回應打架，導致暫時 ID 對齊出問題。
HIGH
在快取更新時，直接靠伺服器配發的 id 來去重複；WebSocket 的資料會帶著真實 ID，對齊時把暫時資料過濾掉。
當使用者在另一台裝置讀過討論串時，未讀數量會卡住不更新。
MED
把 comment_reads 的 upsert 動作廣播到同一個頻道；前端在處理自己的進度時，直接取本地與遠端兩邊的最大值。
在貼上 markdown 時，@ 提及的偵測會誤判（例如 @media 或 @2x）。
LOW
只在寫入資料時，對照工作區成員名單來解析提及對象，並把解析好的使用者 ID 直接存起來，讀取時絕對不再重新解析。

待討論問題

我們要支援編輯，還是只允許刪除後重發？
支援編輯的話，需要多加一個 edited_at 欄位和顯示「已編輯」的 UI。刪除後重發比較簡單，但會丟失原本的回覆錨點。目前傾向在 v1 先做刪除就好。

#312 — 把通知發送搬到 queue 上

Prompt 幫我給審查者寫好 PR #312 的說明。解釋一下動機，帶他們一個檔案一個檔案看過一次， 說清楚每個檔案「為什麼」要這樣改，秀出前後差異，並明確告訴他們審查時要盯緊哪裡。 他們已經半年沒碰通知相關的程式碼了。
一句話重點（TL;DR）

之前發送通知是直接卡在 API 請求的過程中。只要一有流量，這些動作就會讓寫入資料的延遲增加 200–800 毫秒；而且當 SMTP 連線池滿了，信件就會被默默丟掉。這個 PR 把發送工作搬到我們現有的 pg-boss queue 上，讓 API 能馬上回應；萬一發送失敗，也會自動重試。

為什麼要改

上季我們在留言裡加了 @mentions 功能，結果只要有人被 tag 到，同一筆存留言的交易裡就會一口氣觸發三次發送動作（站內通知、email、Slack）。剛上線時這樣搞沒什麼問題。但現在光是一個任務更新，就可能要發送通知給四十個訂閱者，這樣硬幹就不行了。

修改前
  • 發送動作直接卡在寫入資料的 handler 裡
  • SMTP 逾時 = 連帶讓留言也跟著回傳 500 錯誤
  • 沒有重試機制，信丟了就沒了
  • comments.create 的 p99 延遲：1.4 s
修改後
  • Handler 替每個收件者丟一個工作到 queue 裡，然後就直接回應了
  • Worker 會指數退避，自動重試 3 次
  • 重試到底還是失敗的工作，會進到 dead-letter 表裡供人排查
  • comments.create 的 p99 延遲：180 ms（staging 環境數據）

逐一檔案導覽

這裡的排序是依照閱讀順序，不是依照字母排列。先從 worker 開始看，因為這是全新的東西；接著是丟工作進 queue 的呼叫處，最後才是周邊的管線配置。

packages/notify/src/worker.ts 新增 +126

這個 PR 的核心。這是一個 pg-boss 的訂閱器，會把 notify.deliver 工作抓出來處理、解析使用者的頻道偏好，然後呼叫對應的 adapter。重試機制是依照頻道分開設定的，email 可以重試三次，而 Slack 因為它的 API 在我們這邊已經是冪等的了，所以設定成只跑一次。

boss.work('notify.deliver', { batchSize: 20 }, async (jobs) => {
  for (const job of jobs) {
    const { userId, event, channel } = job.data;
    const prefs = await getPrefs(userId);
    if (!prefs[channel]) return;          // 使用者關閉了這個頻道的通知

    try {
      await adapters[channel].send(userId, event);
    } catch (err) {
      if (job.retryCount >= MAX_RETRY[channel]) {
        await deadLetter(job, err);         // 不要 throw 出來，直接 ack 並丟進 dead-letter
        return;
      }
      throw err;                            // 丟出錯誤讓 pg-boss 重新排程
    }
  }
});
packages/api/src/routers/comments.ts 修改 +14 −62

成效展現的地方。本來這個 handler 裡面是直接呼叫 sendEmail、sendSlack 跟 createInApp。現在改成：插入留言、算出收件者，然後把工作丟進 queue。以前那一整坨 try/catch 全都清掉了。

  const comment = await db.comments.insert(input);
  const recipients = await resolveWatchers(input.taskId, input.mentions);

  for (const r of recipients) {
    await sendEmail(r, comment);       // 以前會卡住回應
    await sendSlack(r, comment);
  }
  await boss.insert(recipients.flatMap((r) =>
    CHANNELS.map((ch) => ({
      name: 'notify.deliver',
      data: { userId: r.id, channel: ch, event: toEvent(comment) },
      singletonKey: `${comment.id}:${r.id}:${ch}`,  // 確保冪等
    }))));
  return comment;

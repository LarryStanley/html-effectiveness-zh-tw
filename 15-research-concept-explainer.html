一致性雜湊，只要一個環就搞懂

你有 K 個 key 散在 N 台快取伺服器上。如果某台伺服器掛掉，或是你加了一台新的，會有幾個 key 需要搬家？用最直覺的 hash(key) mod N 算法，答案是「幾乎全都要搬」。但換成一致性雜湊 ，搬家的數量會降到大概 K / N 個。來看看為什麼。

核心手法：把雜湊值映到圓圈上，而不是直線上

把 節點 和 key 都映到同一個 環 上（也就是把雜湊值的輸出空間頭尾相接捲成一圈）。要決定一個 key 歸誰管，就從 key 的位置順時針走下去，碰到的第一個節點就是它的歸宿。當某個節點離開時，只有落在它 弧段 上的 key 需要重新分配給下一個節點，其他 key 一律不動。

有顏色的弧段代表誰擁有那個區間。移除一個節點，就等於把它的弧段交給順時針方向的下一個鄰居；只要不在那個 弧段上的點，顏色都不會變。整個概念就是這樣。

hash mod N一致性雜湊
N 變成 N+1 時要搬家的 key 數量~ (N−1)/N（幾乎全搬）~ 1/(N+1)

你會在哪裡遇到它

只要你需要把狀態分散在一個會變大變小的資源池裡，就會碰到這東西：像快取叢集、分區佇列、物件儲存，或是 帶有黏性會話的請求路由。實務上，正式環境跑的幾乎都是 虛擬節點 版本（讓每個實體節點擁有很多小弧段，而不是一個大弧段 ），因為這樣能把負載打散，重新平衡時也會平順得多。

名詞解釋
環
雜湊函數的輸出範圍，把它想成一個頭尾相接的圓圈，所以 max 的下一格就是 0。
節點
放在環上的伺服器，位置由 hash(node_id) 決定。它擁有自己與逆時針鄰居之間的所有 key。
弧段
節點在環上負責的區段。移除一個節點，就等於把它的弧段併入下一個節點的範圍。
虛擬節點
讓單一實體節點在環上擁有多個位置，這樣每個弧段就會變小，大小也會更平均。
後繼節點
從某個點順時針走，第一個碰到的節點，也就是落在這個點上的 key 的主人。

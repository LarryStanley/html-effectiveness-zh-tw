Debounce 搜尋的三種做法

Prompt 在我們的 React 專案中，針對任務篩選的輸入框，列出三種實作 debounce 搜尋的做法，並比較各自的優缺點。

01直接寫在 useEffect + setTimeout 裡

把 debounce 邏輯直接塞進擁有該輸入框的元件裡。

export function TaskSearch() {
  const [draft, setDraft] = useState('');
  const [query, setQuery] = useState('');

  useEffect(() => {
    const id = setTimeout(() => setQuery(draft), 300);
    return () => clearTimeout(id);
  }, [draft]);

  const { data } = useTasks({ search: query });

  return (
    <input
      value={draft}
      onChange={(e) => setDraft(e.target.value)}
      placeholder="Filter tasks…"
    />
  );
}
優點
缺點
不用多學任何新東西
每個有搜尋功能的地方都要重寫一遍
在 devtools 裡一步一步 trace 很直覺
同一個概念卻要拆成兩個 state
不用動到相依套件或 bundle 大小
延遲時間的常數寫死在元件裡，不好找
Bundle 大小：+0 kb 可測試性：中等 重複使用性：低 SSR 安全：是

02抽出成自訂的 useDebounce hook

把計時器邏輯抽到 src/hooks/ 底下變成共用的 hook。

// src/hooks/useDebounce.ts
export function useDebounce<T>(value: T, ms = 300): T {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const id = setTimeout(() => setDebounced(value), ms);
    return () => clearTimeout(id);
  }, [value, ms]);
  return debounced;
}

// TaskSearch.tsx
const [draft, setDraft] = useState('');
const query = useDebounce(draft, 300);
const { data } = useTasks({ search: query });
優點
缺點
只要 import 一次，篩選器、命令列、看板搜尋都能用
多一個檔案要維護和寫文件
用 fake timer 跑單元測試很簡單
泛型 T 會讓意圖稍微沒那麼明確
延遲時間是清楚的參數，方便調整
每次打字還是會觸發重新渲染
Bundle 大小：+0.2 kb 可測試性：高 重複使用性：高 SSR 安全：是

03直接用輕量的外部套件

引入 use-debounce 來同時處理值和 callback。

import { useDebouncedCallback }
  from 'use-debounce';

export function TaskSearch() {
  const [query, setQuery] = useState('');

  const onChange = useDebouncedCallback(
    (next: string) => setQuery(next),
    300,
    { leading: false, maxWait: 1000 },
  );

  const { data } = useTasks({ search: query });

  return (
    <input
      defaultValue=""
      onChange={(e) => onChange(e.target.value)}
    />
  );
}
優點
缺點
leading、trailing、maxWait 都幫你處理好了
多了一個執行階段的相依套件要審查和更新
用 callback 形式可以跳過中間不必要的重新渲染
uncontrolled input 跟 Acme 現有的表單模式不一致
邊界情境都測過了（unmount、flush、cancel）
gzip 後大概 1.4 kb，其實自己寫就有了
Bundle 大小：+1.4 kb 可測試性：高 重複使用性：高 SSR 安全：是

建議做法

選 做法 02，抽出自訂的 useDebounce hook。 Acme 現在已經有三個地方各自手寫了同樣的 inline 邏輯 （任務篩選器、命令面板、成員選擇器），所以抽出一個共用的 hook 就能消除重複的程式碼，又不用多裝一個相依套件。

除非之後真的用到 maxWait 或 flush() 這類功能， 否則不用考慮做法 03。等到需求多到十行 code 的 hook 搞不定時， 這個套件才有它引入的價值。

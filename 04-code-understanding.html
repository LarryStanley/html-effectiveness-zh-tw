整個專案庫的驗證流程是怎麼跑的

Acme 是用 cookie-based session：瀏覽器從頭到尾都不會拿到 bearer token。每個經過驗證的請求都會打到位於 /api/* 的端點，統一過 verifyToken() 這一層中介層，最後在 req.ctx 裡放進一筆 Session 資料讓後續的 handler 直接用。全專案裡，只有這層中介層會去碰 session store，而 session store 則是全專案唯一會去讀寫 sessions 資料表的地方。也就是說，整個信任邊界就這麼一處，很好管理。

src/app/providers/AuthProvider.tsx :22-48

這個 React provider 一掛載，就會直接發一個 GET /api/session，而且帶上 credentials: 'include'，讓 fw_sid cookie 可以跟著送出去。接著看回應內容，把 currentUser 塞進 context 裡；如果沒拿到資料就維持 null，讓 router 直接跳到登入畫面。

顯示原始碼 
// src/app/providers/AuthProvider.tsx
export function AuthProvider({ children }: Props) {
  const [user, setUser] = useState<User | null>(null);

  useEffect(() => {
    fetch('/api/session', { credentials: 'include' })
      .then(r => r.ok ? r.json() : null)
      .then(setUser);
  }, []);

  return <AuthCtx.Provider value={{ user }}>{children}</AuthCtx.Provider>;
}

3
src/middleware/auth.ts :14-31

這裡就是系統的信任邊界。verifyToken 會去讀取已經簽章的 fw_sid cookie，交給 SessionStore 去查對應的 session 資料，查到就塞進 req.ctx.session，查不到就直接回 401。整個專案裡所有受保護的 route 都會掛這個函式，所以只要改這裡的行為，就等於一次改動全專案的驗證邏輯。

顯示原始碼 
// src/middleware/auth.ts
export async function verifyToken(req, res, next) {
  const raw = req.signedCookies['fw_sid'];
  if (!raw) return res.status(401).end();

  const session = await SessionStore.get(raw);
  if (!session || session.expiresAt < Date.now()) {
    return res.status(401).end();
  }

  req.ctx = { session };
  next();
}

src/lib/sessionStore.ts :8-52

SessionStore 其實就是一個簡單的 read-through 快取：它會先查 in-process 的 LRU，查不到再去 Postgres 撈。寫入操作（像是 create 或 revoke）則是直接寫進資料庫，同時讓快取失效，確保其他 worker 不會繼續吐過期的 session 資料。

顯示原始碼 
// src/lib/sessionStore.ts
const cache = new LRU<string, Session>({ max: 5000, ttl: 60_000 });

export const SessionStore = {
  async get(id: string) {
    const hit = cache.get(id);
    if (hit) return hit;
    const row = await db.one(SELECT_SESSION, [id]);
    if (row) cache.set(id, row);
    return row ?? null;
  },
  /* create, revoke, touch ... */
};

db/migrations/004_sessions.sql :1-18

sessions 資料表的 primary key 是隨機產生的 32-byte id（也就是 cookie 的值），並且在 user_id 上建了索引，方便用來做「登出所有裝置」這類操作。過期時間一方面在這裡靠 expires_at 把關，另一方面在中介層也會再檢查一次，算是雙重保險。

顯示原始碼 
-- db/migrations/004_sessions.sql
create table sessions (
  id          text primary key,
  user_id     uuid not null references users(id),
  created_at  timestamptz default now(),
  expires_at  timestamptz not null,
  ip          inet,
  user_agent  text
);
create index sessions_user_id_idx on sessions(user_id);

關鍵檔案

  • src/middleware/auth.ts 請求驗證的統一入口。
  • src/lib/sessionStore.ts LRU 加 Postgres 的 session 查詢；全專案唯一去碰資料庫的地方。
  • src/server/routes/session.ts 把目前的 session 回傳給前端。
  • src/server/routes/login.ts 透過 SessionStore.create 把憑證換成 cookie。
  • src/app/providers/AuthProvider.tsx 前端的 context，用來同步 server 端的 session 狀態。
  • db/migrations/004_sessions.sql sessions 資料表與索引的 schema。

陷阱與眉角

  • SessionStore 裡的 LRU 是每個 process 各自一份。所以當你 revoke 掉某個 session 時，只會清掉當前 process 的快取，其他 worker 最長可能還是會繼續吐舊資料，直到 60 秒後 TTL 到期才會失效。
  • verifyToken 雖然是拿 expiresAt 跟 Date.now() 比大小，但資料庫欄位其實是 timestamptz 型別。資料庫 driver 回傳的時候已經是 Date 物件了，所以直接比沒問題；但如果你之後想重構，千萬不要隨便把它改成純字串比對，除非你連檢查邏輯也一起調整。

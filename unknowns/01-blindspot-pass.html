未知盲點：Acme auth 模組

掃描了 services/auth/（47 個檔案），從 2025 年 1 月到現在共有 214 個 commit 動過這裡，目前有 3 個遷移工程在跑，還有 2 次被撤銷的嘗試。以下是你遲早會踩到的七個雷。

你以為的需求

「加一個新的 SSO provider」，聽起來就是：實作一個介面、註冊一下、寫好導向的 handler，收工。如果是在架構乾淨的專案，大概兩天就能搞定。

你實際上會走進什麼樣的坑

這個模組正在遷移 session 儲存方式，而且有一個 provider 偷偷繞過了大家以為有跑的中介層。之前有人試過做這件事，結果被 revert，再加上一套從 provider 介面根本看不出來的身分綁定邏輯。

盲點列表
地雷 01

Session 被寫了兩份 — 你最先找到的 Redis store 其實不對

Acme 正在把 session 從 Postgres 遷移到 Redis（MIG-118，三月之後就卡住了）。現在每次寫入 session 都會經過 SessionBridge.write()，這個方法會同時寫進 RedisSessionStore 和舊的 pg_sessions 資料表。但遷移還沒做完，所以讀取還是從 Postgres 來。

為什麼會踩雷你搜尋「session」會先看到 RedisSessionStore，它看起來最正規、程式碼最新、文件也最齊全。如果你直接寫進去，在 dev 環境登入看起來會很正常（因為 dev 有開 Redis 讀取），但到了 prod 卻會默默失敗，因為 prod 還是從 Postgres 讀資料。
在幫新 provider 建立 session 時，所有操作都要經過 SessionBridge，絕對不要直接碰 RedisSessionStore 或 pg_sessions。在寫任何 session 相關程式碼前，先解釋 MIG-118 的讀寫分離機制。
地雷 02

SAML provider — 你最直覺想抄的範本 — 其實繞過了 auth 中介層

除了 SAML 以外，每個 provider 都掛在 authPipeline 下面，這個 pipeline 會處理速率限制、稽核 log 和裝置信任檢查。但 saml/handler.ts 把它的 ACS 端點直接綁在 router 上，因為 2024 年時 SP 發動的 POST callback 搞壞了 CSRF 那層，後來也沒人回去收尾。程式碼裡還留著一個 14 個月前的 // TODO(priya): fold back into pipeline。

為什麼會踩雷SAML 是最近才加進來的 provider，所以你一定會想直接抄它的檔案。一旦抄了，你的新 provider 上線後就會沒有速率限制、也沒有稽核 log。更麻煩的是，這不會讓任何測試失敗，因為合約測試根本沒有檢查中介層是否存在。
用 oauth/google.ts 作為架構範本，不要用 saml/handler.ts，因為 SAML 刻意繞過 authPipeline。確認新 provider 的路由是掛在 authPipeline 裡面，並把掛載點秀給我看。
歷史教訓 03

已經有人做過這件事了 — 而且被撤銷的原因現在依然成立

PR #2841（「新增 Okta OIDC provider」，2025 年 9 月）合併後九天就被撤銷了。revert 的 commit 說明了原因：Acme 的 callback URL 必須帶有租戶後綴（/auth/callback/:provider/:workspaceSlug），這樣工作區範圍的邀請連結才能在導向過程中存活下來。但 Okta 嚴格比對導向 URI，拒絕了這個動態後綴，導致受邀使用者登入後跑錯工作區。當時大家達成共識的解法是用 AUTH_STATE_SECRET 簽署 state 參數的 workspace 聲明，這個方案在 revert 的討論串裡設計好了，卻從來沒被實作出來。

為什麼會踩雷現在的程式碼裡完全看不出這件事。你可能會重新實作 #2841 的做法，通過所有本地測試（因為 dev 環境只有單一工作區），然後等到第一個受邀使用者登入時，才在正式環境重新發現這個 bug。
先讀過 PR #2841 的撤銷紀錄。實作它提議的簽署 state 參數 workspace 聲明，不要用租戶後綴的 callback URL，並針對工作區邀請流程補上測試。
缺少的概念 04

身分和帳號是兩回事 — 但 provider 介面把這件事藏起來了

Acme 把 users 和 identities 拆開：一個帳號可以綁定多個身分（帳密、Google、SAML），它們在 identities 資料表裡關聯起來。provider 介面只會回傳 profile，而綁不綁定的決策其實發生在 identity/linker.ts 裡，靠的是已驗證的 email 來判斷。如果某個 provider 無法保證 email 已驗證（很多 SSO IdP 都不保證），就必須設定 requiresManualLink: true，否則 linker 會很開心地共用公司郵件別名的兩個不同的人合併在一起。

為什麼會踩雷如果你照著介面直覺實作，一切看起來都很正常。直到某天客戶的 IdP 傳來一個未驗證或被重新分配的 email，你的 provider 就會默默把陌生人的身分綁到現有帳號上。這是帳號被接管的資安漏洞，不是單純的登入 bug。
在寫程式碼前，先帶我走過 identity/linker.ts 的邏輯。明確判斷這個 IdP 是否保證 email 已驗證；如果不保證，就設定 requiresManualLink，並做好手動綁定的確認流程。
地雷 05

Token refresh 被功能旗標控制，dev 開著但 prod 關著

Refresh token 的輪換機制被 flags.yaml 裡的 auth.refresh_rotation 功能旗標（feature flag）控制。dev 和 staging 預設是 true，prod 則是 false（因為五月發生過時鐘誤差事件後暫停推出）。旗標關閉時，prod 的 session 只能活 ACCESS_TOKEN_TTL（12 小時），時間一到就直接失效，根本沒有 refresh 的路徑。

為什麼會踩雷你寫的任何 refresh 邏輯在 dev 環境測試都沒問題，但到了 prod 就成了沒用的死程式碼。更糟的是，如果你的 provider 假設會發生 refresh，prod 使用者每 12 小時就會在操作途中被登出，然後 bug 回報只會寫「SSO 莫名其妙把我登出了」，矛頭全指向你的程式碼。
針對 auth.refresh_rotation=false（prod 的實際情況）來實作。讓 provider 在 12 小時硬性過期的情況下也能正常運作，並另外註明如果輪換機制恢復上線會有什麼變化。
慣例 06

註冊一個 provider 需要三個步驟，但只有兩個查得到

Provider 通過編譯不代表它就能用了。你必須 (1) 在 providers/registry.ts 註冊，(2) 透過 migration 把資料寫入 auth_providers 資料表（因為管理介面是讀資料庫，不是讀 registry），還要 (3) 把它加進 __tests__/provider-contract.spec.ts 的測試清單裡，這份測試會對每個註冊的 provider 跑共享的合約測試（狀態驗證、重送保護、登出擴散）。如果 registry 和測試清單不一致，CI 就會直接報錯。

為什麼會踩雷如果你漏掉了 migration，provider 可以編譯、可以通過測試，但就是不會出現在任何工作區的 SSO 設定裡。已經有兩個工程師分別在這上面各浪費了半天，而且沒有任何文件提過這件事。
三個註冊步驟全都要做：registry.ts、一個 auth_providers 的 migration，以及 provider-contract 的測試清單。在我們認定完成之前，先讓合約測試針對新 provider 跑出全綠燈的結果給我看。
地雷 07

登出不是一個 function call — 它是一個事件，而且 review 播放器在監聽

當 session 被撤銷時，系統會在內部事件匯流排上發布 auth.session.revoked。即時審查播放器握有自己的長效播放權限，只有當它收到那個事件時才會釋放權限。logout.ts 已經幫現有的 provider 處理好這件事了，但是 SSO 的單一登出（SLO）callback 是走另一條路徑進來的，必須由它們自己發布這個事件。

為什麼會踩雷如果你漏掉了這個事件，登出看起來「成功」了（session 紀錄確實刪掉了），但任何還開著審查分頁的人，在權限 TTL 過期前（最長 24 小時）都還能繼續播放私人影片。對於一家把審查流程賣給工作室的公司來說，這是資安事件，不是普通的 bug。
處理 SLO：新 provider 裡的任何撤銷路徑都必須發布 auth.session.revoked 事件。加上一個測試，確保 IdP 發動登出時，審查播放器的播放權限會被釋放。

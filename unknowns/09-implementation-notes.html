偏離計畫 2 / 415:22

Queue 把 Date 物件轉成字串再傳回來

原本的計畫
ExportJobPayload.requestedAt: Date，直接傳給 worker，再跟 review 的 updated_at 比對，用來偵測過期的匯出請求。
程式碼實際狀況
BullMQ 會把 payload 做 JSON 序列化後存進 Redis，所以 worker 拿到的是 ISO 字串，不是 Date。在單元測試裡，這個新舊比對偷偷做了型別轉換還能勉強過關，但在整合測試跑重新投遞的任務時就出錯了。
保守做法
把 payload 型別定義改成傳輸格式（requestedAt: string ISO-8601），在 exportConsumer.ts 的 worker 邊界統一解析一次。不另外寫自訂序列化器。
之後可以討論
其他 media-tasks 的 payload 八成也有這種潛在 bug，如果幫 job payload 寫一個共用的 zod codec，應該能一次解決這類問題。

偏離計畫 4 / 416:20

訪客審查者能看 review，但沒辦法下載素材

原本的計畫
匯出權限 = review 層級的 ACL：只要能打開 review 的人就能匯出。
程式碼實際狀況
Workspace 層級的 guest_reviewer 角色雖然有 review 權限，但 can_download_assets = false。這個設計是讓外部關係人可以在不把素材帶出去的情況下留言。如果只檢查 review 的 ACL，訪客就能透過匯出功能把原始影片弄走。
保守做法
所有匯出格式遇到 guest_reviewer 都直接回傳 403，跟 assetAccess.ts 裡現有的素材下載檢查保持一致。這是最嚴格也站得住腳的做法。

這對下一次嘗試有什麼影響

底下這三點可以直接貼進下一次的計畫或提示詞裡，免得下一輪又重新踩到今天的雷。

  1. 一開始就先講明舊資料的限制：大約 12% 的標註（migration 0087 之前建立的）沒有 frame_ts。該推算時間還是做成附帶檔案，這個決定該在實作開始前就定案，而不是在 14:41 開發到一半才來想。
  2. 把 queue payload 明確定義為傳輸格式：BullMQ 會透過 JSON 序列化，所以 payload 介面應該全用 ISO 字串，不要用 Date。如果幫 media-tasks 的 payload 寫個共用的 zod codec，就能徹底解決這類 bug。
  3. 加入相依套件前，先盤點現有工具：zipStream.ts 跟 useProgressChannel() 早就有了，而且都比計畫裡的做法更好。可以在計畫範本里加一步「搜尋看看有沒有現成的東西」，並且在下一個動到 ACL 的功能開工前，先敲定訪客審查者的匯出權限。

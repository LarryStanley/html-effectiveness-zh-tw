偏離 2 / 415:22

佇列將 Date 物件以字串形式往返傳遞

計畫的內容
ExportJobPayload.requestedAt: Date，傳遞給 worker 並與評論的 updated_at 進行比較，以偵測過時的匯出。
程式碼顯示的狀況
BullMQ 將承載資料 JSON 序列化進 Redis，因此 worker 收到的是 ISO 字串，而非 Date。過時比較在單元測試中悄悄地發生了隱含轉換並通過，但在整合測試的重新傳遞工作中失敗了。
保守的選擇
將承載資料型別定義為傳輸格式（requestedAt: string ISO-8601），並在 exportConsumer.ts 的 worker 邊界統一解析一次。不使用自訂序列化器。
待後續討論
其他 media-tasks 的承載資料可能也有相同的潛在錯誤。為工作承載資料加入一個共享的 zod 編解碼器，應該能解決這整類問題。

偏離 4 / 416:20

訪客審查者可以檢視評論，但無法下載資產

計畫的內容
匯出權限 = 評論層級的存取控制清單（ACL）：任何能開啟評論的人都能匯出它。
程式碼顯示的狀況
工作區層級的 guest_reviewer 角色授予了評論存取權，但 can_download_assets = false。客戶端使用這個角色讓外部關係人發表評論，同時無法將媒體帶離站外。如果只檢查評論的 ACL，訪客就能透過匯出功能竊取來源影片。
保守的選擇
所有格式都對 guest_reviewer 回傳 403，這對應了 assetAccess.ts 中現有的資產下載檢查。這是最嚴格且合理的認定。
待後續討論
理論上訪客還是應該能拿到 CSV（只有標註，沒有媒體）。但這是產品決策，不歸我管，已記錄在下方。

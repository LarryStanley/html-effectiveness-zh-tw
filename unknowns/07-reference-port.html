來源 vendor/rate-limiter (Rust, 612 loc, 3 個模組) 目標 acme-web 專案裡的 src/api/retry/ 讀取 bucket.rs · backoff.rs · budget.rs · 14 個測試
1 這個 crate 到底做了什麼
  • 權杖桶（token bucket）許可機制（bucket.rs）：每送出一個請求，就會從桶子裡扣掉 1 個 token。桶子會以 rate_per_sec = 8 的速率回補，上限是 burst_capacity = 40。桶子一開始是全滿的，所以全新的 client 可以瞬間打出 40 個請求。
  • 延遲回補且無條件捨去：token 是等請求進來時才根據經過的奈秒數（經過時間 × 速率 ÷ 10⁹）現算的。小數點後的 token 一律不存起來，而是靠 last_refill 記錄時間：只有真的湊到一個完整的 token 時，時間才會往前推進。這樣一來，還沒湊滿一個 token 的剩餘時間就不會白白損失掉。
  • 去關聯抖動（decorrelated jitter）退避（backoff.rs）：遇到可以重試的錯誤時，下一次的延遲時間會從 [base, min(prev × 3, cap)] 這個區間隨機抽取，其中 base = 250ms，cap = 30s。每次抽出的值都會變成下一次的種子。這會讓延遲時間隨機亂飄，而不是死板板地每次都乘兩倍，好處是發生大當機時，所有 client 的重試時間才不會全部擠在一起。
  • 重試預算（budget.rs）：這是另一個獨立的桶子。第一次就成功的請求會存入 1 個單位；每發生一次重試，就會扣掉 10 個單位。如果餘額不夠扣，系統就會直接拒絕重試（但第一次發出的請求不受影響）。這樣一來，就算伺服器持續大當機，重試造成的流量也會被壓在 10% 左右。
  • 錯誤分類是呼叫端的事：這個 crate 完全不碰錯誤內容。它只提供 acquire()、next_delay()、try_withdraw()，至於什麼錯誤可以重試，交給呼叫端自己決定。TypeScript 版本也會守住這條界線：Acme 在 src/api/errors.ts 裡的 isRetryable() 依然是判斷的唯一標準。

    (elapsedMs * this.ratePerSec) / 1000

1系統時鐘倒退。 saturating_duration_since 會把負數的經過時間硬壓成 0。在 NTP 校時下，Date.now() 有可能往回跳；移植版改用單調遞增的 performance.now()，而且還是保留了 Math.max(0, …) 做為雙重保險。
2整數與浮點數的差異。 Rust 的 u64 除法會自動截斷；JS 的除法不會。所以這裡用 Math.floor 來還原截斷行為。把奈秒換成毫秒是安全的：在 rate = 8 的情況下，elapsedMs * 8 遠低於 2⁵³，不用擔心精密度流失。
3這個判斷是核心。 last_refill 只有在真的湊到一個完整 token 時才會更新。如果自作主張把這個判斷拿掉（很多人會覺得這樣「比較簡潔」），每次呼叫時還沒湊滿一個 token 的進度就會被默默丟掉。在低速率的情況下，如果頻繁輪詢，桶子就會永遠補不滿。這個邏輯被原封不動保留了下來，而且還補了一支回歸測試。

nextDelay(): number {
  const hi = Math.min(this.prevDelayMs * 34, this.capMs);
  const lo = this.baseMs;
  const span = Math.max(hi, lo) - lo;
  const ms = lo + Math.floor(
    this.random() * (span + 1)
  )5;
  this.prevDelayMs = ms;6
  return ms;
}
4飽和乘法。 saturating_mul(3) 是為了防止 u64 溢位。在 JS 裡根本不會溢位，因為 capMs = 30_000 早就把乘積限制在 2⁵³ 以下，所以這個保護機制被刻意拿掉了（詳見 §3 的「捨棄」欄位）。
5閉區間與開區間。 Rust 的 lo..=hi 包含頭尾兩個邊界值。如果直接寫成 lo + random() * (hi - lo)，永遠抽不到 hi。在 Math.floor 裡面加 + 1 就能還原閉區間。這一個字元的差異超容易出錯，所以我特地標出來，讓你決定要不要買單。
6保存狀態的種子。 每次抽出的延遲時間，都會變成下一次計算的 prev。這正是它被稱為去關聯抖動的原因，而不是單純加上抖動的指數退避。成功時呼叫 reset() 會把 prevDelayMs 還原成 baseMs，這跟 crate 裡 Backoff::reset 的行為一模一樣。this.random 被設計成可以注入的，方便做確定性的測試（crate 在測試時用的是帶種子的 SmallRng）。

tryWithdraw(): boolean {
  // 沒有 lock：單一執行緒的事件迴圈7 —
  // 但檢查與扣款之間絕對不能有 await。
  this.depositDrip(this.clock());

    this.balance -= WITHDRAW_COST;8
    return true;
  }
  return false9;

7從執行緒安全到事件迴圈。 Crate 裡會用 Mutex，是因為 Rust 的呼叫端可能會有好幾個 worker 執行緒在扣款。Acme 的 client 跑在單一事件迴圈上，所以鎖可以拿掉。不過，當初那個鎖保證的「不可分割性」，還是得靠規矩來維持：tryWithdraw 從頭到尾都是同步執行的。我會在 eslint 加一條 no-await-in-budget 邊界註解，並且寫一個測試，確保這個方法永遠不會回傳 Promise。
8檢查與扣款必須連在一起。 兩邊的結構是一樣的。這裡最怕的不是資料競態，而是未來有人改程式時，不小心在檢查跟扣款中間塞了一個 await（比如說想加 log），結果讓兩個還在飛行中的重試同時通過檢查。
9直接拒絕，不要排隊。 預算用完時，crate 會直接回傳 false。這個請求不是當作第一次發送出去，就是直接失敗。移植版絕對不能「雞婆」地把重試排到後面再送；如果這樣做，就等於把這個預算機制想要擋下的重試風暴又給請回來了。
3 行為：完整保留 / 刻意調整 / 捨棄不用
完整保留
  • 回補的捨去機制與判斷 — 只認完整的 token；lastRefill 只有在真的生出 token 時才更新
  • 抖動公式 — 在 [base, min(prev×3, cap)] 區間均勻分佈，頭尾都包含在內
  • 預算的會計規則 — 第一次成功 +1，重試 -10，上限 1000
  • 桶子一開始是全滿的 — 可以瞬間打出 40 個請求是刻意的（跟 crate 的測試 burst_at_t0 對應）
  • 錯誤分類交給呼叫端 — 限制器完全不碰錯誤內容
刻意調整
  • Instant → performance.now() — 兩者都是單調時鐘；在速率 8/s 下毫秒解析度夠用了
  • u64 奈秒 → number 毫秒 — 所有乘積都保證小於 2⁵³；靠 Math.floor 模擬整數除法
  • Mutex<Budget> → 一般欄位 — 靠「全同步」的規矩與測試來確保不可分割性
  • SmallRng → 注入式的 random() — 預設用 Math.random，測試時則注入帶種子的假物件
捨棄（用不到）
  • saturating_mul 溢位保護 — 換成毫秒範圍並套用上限後，根本走不到這段
  • Send + Sync 實作與 Arc 複製 — 沒有多執行緒需要共享
  • tokio / async-std 功能旗標（feature flag） — 移植版天生就不綁特定執行階
  • Prometheus 計數器 — Acme 改用 telemetry.track() 處理；掛鉤點保留下來

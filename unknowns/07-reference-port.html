來源 vendor/rate-limiter (Rust, 612 loc, 3 modules) 目標 acme-web 專案中的 src/api/retry/ 讀取 bucket.rs · backoff.rs · budget.rs · 14 個測試
1 這個 crate 實際做了什麼
  • 權杖桶（token bucket）准入控制（bucket.rs）：每個發出的請求都會從桶子裡扣除 1 個權杖，桶子以 rate_per_sec = 8 的速率回補，上限為 burst_capacity = 40。桶子一開始是全滿的，所以全新的用戶端可以瞬間送出 40 個請求。
  • 延遲回補且整數截斷：只有在請求到達時，才會根據經過的奈秒數 × 速率 ÷ 10⁹ 來產生權杖。小數點後的權杖永遠不會被存下來，last_refill 只在至少產生一個完整權杖時才會更新，所以不足一個權杖的剩餘時間會累積到下次計算。
  • 去相關抖動退避（backoff.rs）：遇到可重試的失敗時，下一次的延遲會從 [base, min(prev × 3, cap)] 区間內均勻抽取，其中 base = 250ms，cap = 30s。每次抽取的值都會成為下一次的種子，延遲時間會隨機遊走而不是整齊倍增，這能在服務中斷後打散用戶端的同步重試。
  • 重試預算（budget.rs）：這是第二個獨立的桶子。每次第一次就成功的請求會存入 1；每次重試都會扣除 10。當餘額不足以支付扣除時，重試會被全面拒絕（第一次的請求仍會發出）。這能在持續中斷期間，將重試放大比例限制在約 10%。
  • 失敗分類是呼叫端的工作：這個 crate 完全不檢查錯誤。它只暴露 acquire()、next_delay()、try_withdraw()，由呼叫端決定哪些算可重試。TS 移植版會保留這個界線，Acme 在 src/api/errors.ts 裡的 isRetryable() 仍然是唯一的事實來源。

1時鐘倒退問題。 saturating_duration_since 會將負的經過時間限制為零。Date.now() 在 NTP 校時下可能會往回跳；移植版使用了單調的 performance.now()，而且保留了 Math.max(0, …) 做為雙重保險。
2整數與浮點數運算。 Rust 的 u64 除法會截斷；JS 的除法不會。Math.floor 還原了截斷行為。使用毫秒而非奈秒是安全的：在 rate = 8 時，elapsedMs * 8 遠低於 2⁵³，不會有精度損失。
3承載關鍵邏輯的條件判斷。 last_refill 只有在產生完整權杖時才會更新。如果拿掉這個判斷（很多人會以為這是「簡化」），每次呼叫都會默默丟棄不足一個權杖的進度。在低速率下，如果頻繁輪詢，桶子就永遠不會回補。這點已完整保留，並加上回歸測試。

4飽和乘法。 saturating_mul(3) 防止了 u64 溢位。在 JS 中這不會溢位，因為 capMs = 30_000 早就把乘積限制在 2⁵³ 以下了，所以這個保護機制被刻意拿掉了（見 §3 的「捨棄」欄位）。
5包含與不包含邊界的區間。 Rust 的 lo..=hi 包含兩個邊界。單純的 lo + random() * (hi - lo) 永遠產生不了 hi。Math.floor 裡的 + 1 還原了包含邊界的行為。這一個字元的差異很容易產生 bug，特別標示出來讓你決定是否接受。
6具狀態的種子。 每次抽取的值都會成為下一次迭代的 prev，這正是讓它去相關的原因，而不是單純的指數退避加抖動。reset() 在成功時會將 prevDelayMs = baseMs 還原，與 crate 的 Backoff::reset 一致。this.random 採用注入方式，方便在測試中保證確定性（crate 在測試中使用帶種子的 SmallRng）。

7執行緒安全 → 事件迴圈。 這個 crate 之所以存在 Mutex，是因為 Rust 的呼叫端會從工作執行緒扣除額度。Acme 的用戶端在單一事件迴圈上執行，所以鎖就不見了，但它提供的原子性必須透過慣例來保留：tryWithdraw 徹頭徹尾都是同步的，而且我會加上 eslint 的 no-await-in-budget 邊界註解，並寫一個測試確保這個方法永遠不回傳 Promise。
8檢查與扣除保持一體。 兩邊的結構相同。這裡的失效模式不是資料競態，而是未來的重構可能會在檢查與扣除之間插入 await（例如為了記錄 Log），導致兩個進行中的重試都通過檢查。
9拒絕，絕不排隊。 預算用盡時，crate 會立即回傳 false，請求不是做為第一次嘗試發出，就是立刻失敗。移植版絕對不能「好心」地把重試排到稍後執行；這樣做會重建出這個預算機制原本就是要防止的重試風暴。
3 行為：完整保留 / 刻意變更 / 捨棄
完整保留
  • 回補截斷與條件判斷 — 僅限完整權杖；lastRefill 只在產生權杖時更新
  • 抖動公式 — 在 [base, min(prev×3, cap)] 區間內均勻分佈，兩端皆包含
  • 預算經濟學 — 第一次成功 +1，重試 −10，上限 1000
  • 桶子一開始全滿 — 40 個請求的冷啟動爆發是刻意的（符合 crate 的測試 burst_at_t0）
  • 由呼叫端分類錯誤 — 限制器絕不檢查失敗
刻意變更
  • Instant → performance.now() — 兩者皆為單調時鐘；在 8/s 的速率下毫秒解析度已經足夠
  • u64 奈秒 → number 毫秒 — 所有乘積皆可證明 < 2⁵³；Math.floor 重現整數除法
  • Mutex<Budget> → 純欄位 — 透過僅限同步呼叫的慣例與測試來保證原子性
  • SmallRng → 注入的 random() — 預設為 Math.random，在測試中使用帶種子的虛擬實作
捨棄（不需要）
  • saturating_mul 溢位保護 — 在毫秒範圍內套用上限後就永遠不會觸發
  • Send + Sync 實作與 Arc 複製 — 沒有跨執行緒共用的需求
  • tokio/async-std 功能旗標（feature flag） — 移植版在結構上與執行環境無關
  • Prometheus 計數器 — Acme 改為透過 telemetry.track() 發送；但保留了掛載點

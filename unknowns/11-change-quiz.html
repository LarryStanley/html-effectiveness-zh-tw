心智模型

一張圖勝過 14 個檔案：現在和以前相比，匯出請求到底怎麼跑的。

在這次變更之前，「匯出片段」代表瀏覽器要用 MediaRecorder 在客戶端把影格串起來，然後上傳結果。這樣做很慢、會把分頁卡住，而且在 Safari 上直接壞掉。現在客戶端只要發出匯出請求就好；新的 worker 會從原始媒體生出結果，客戶端再去輪詢工作狀態，等到出現一個帶簽章的下載 URL 就搞定了。

這次引入的三個非直觀行為

這些是隨便掃過 diff 看不出來的眉角。每一個都是刻意設計的，以下附上原因。

01

匯出是從原始上傳檔案算繪的，不是審查者看的那個代理檔案

內容Worker 會抓 media/originals/，絕對不會拿 720p 的審查代理檔案。因此，匯出出來的片段看起來會比審查者畫標記時看到的更清晰。
原因剪輯師匯出片段是為了交給客戶，如果給人家壓縮過的代理檔案就太丟臉了。標記座標是用常規化（0–1）格式儲存的，所以可以正確重新投影到全解析度的影格上。
位置worker/export/render.ts:41 worker/export/burn_in.ts:88
02

匯出工作在 worker 崩潰後依然存活，靠的是可視度逾時，不是重試機制

內容系統會透過設定 locked_until = now() + 10min 來認領工作。如果 worker 在算繪過程中陣亡，並不會觸發重試；鎖定到期後，下一個有空檔的 worker 會從頭接手這項工作。
原因ffmpeg 的算繪過程並不是冪等的，而且我們目前不想處理重試佇列需要的寄不出去信件（dead-letter）邏輯。鎖定到期機制只用一個欄位，就做到了「至少送達一次」的語意。代價是：崩潰的工作在恢復前，會看起來「處理中」長達 10 分鐘。
位置db/migrations/0142_export_jobs.sql worker/export/claim.ts:19
03

下載 URL 在 24 小時後失效，但匯出檔案本身會保留 7 天

內容GET /api/exports/:id 回傳的 S3 簽章 URL 有效期只有 24 小時，但底層的物件在 7 天的生命週期規則觸發前都不會被刪除。重新呼叫這個端點會產生一組新的 URL，不需要重新算繪。
原因較短的 URL 有效期可以限制連結被轉發到工作區外（匯出檔案可能包含未公開的畫面）；較長的物件保留時間意味著，一個過期的 Slack 連結只需要花一次 API 呼叫就能救回來，不需要重新花 90 秒算繪。
位置api/exports/get.ts:57 infra/s3_lifecycle.tf:23
依賴的現有行為

以工作區為範圍的簽章 URL 驗證。 GET /api/exports/:id 本身沒有做任何權限檢查，它直接沿用 requireWorkspaceMember 這個中介層，這個中介層本來就把守著所有媒體相關的路由。如果這個中介層的 session 處理邏輯將來改變了（現在剛好有一張關於訪客審查者 session 的 ticket，BL-2214），匯出下載的行為就會跟著變。而這份 diff 裡面沒有任何東西會提醒你這件事。

心智模型

一張圖勝過 14 個檔案：現在和以前相比，匯出請求的流程長什麼樣。

在這次變更之前，「匯出片段」意味著瀏覽器使用 MediaRecorder 在客戶端逐幀拼接畫面，然後上傳結果。這樣做不僅速度慢、會把瀏覽器分頁卡死，在 Safari 上還會出問題。現在，客戶端只需要發起匯出請求；新的 worker 會從原始媒體檔案在伺服器端進行算圖，客戶端則透過輪詢工作任務，直到取得有效的下載 URL。

這次引入的三個非顯而易見行為

這些是隨便看看 diff 不會察覺的部分。每一項都是刻意為之，以下附上原因。

01

匯出是從原始上傳檔案算圖，而非審查者觀看的代理檔案

內容Worker 會提取 media/originals/，絕不使用 720p 的審查代理檔案。因此，匯出的片段看起來可能會比審查者畫標記時看到的畫面更清晰。
原因剪輯師匯出片段是為了交給客戶；如果送出壓縮過的代理檔案會很丟臉。標記座標是以正規化（0–1）方式儲存的，因此能正確重新投影到原始解析度的畫面上。
位置worker/export/render.ts:41 worker/export/burn_in.ts:88
02

匯出工作任務能在 worker 崩潰後存活，靠的是可視度逾時，而非重試

內容系統會透過設定 locked_until = now() + 10min 來認領工作任務列。如果 worker 在算圖過程中掛掉，不會觸發任何重試；鎖定一旦到期，下一個閒置的 worker 就會從頭接手該任務。
原因ffmpeg 的算圖過程並非冪等，而且我們目前不想導入需要無效信件佇列處理的重試機制。鎖定到期機制僅用一個欄位就實現了「至少一次」的語意。副作用是：崩潰的工作任務看起來會卡在「處理中」狀態長達 10 分鐘，之後才會恢復。
位置db/migrations/0142_export_jobs.sql worker/export/claim.ts:19
03

下載 URL 會在 24 小時後失效，但匯出檔案本身會保留 7 天

內容GET /api/exports/:id 回傳的 S3 簽署 URL 有效期為 24 小時，但底層的物件直到 7 天的生命週期規則觸發前都不會被刪除。再次呼叫該端點會產生新的 URL，而不需要重新算圖。
原因較短的 URL 有效期可以限制連結在工作區外被轉發（因為匯出內容可能包含未發布的片段）；較長的物件存活時間則意味著，如果一個過期的 Slack 連結需要恢復，只需花費一次 API 呼叫，而不需要重新算圖 90 秒。
位置api/exports/get.ts:57 infra/s3_lifecycle.tf:23
依賴的現有行為

工作區範圍的簽署 URL 驗證。 GET /api/exports/:id 本身並沒有做任何權限檢查，而是直接沿用 requireWorkspaceMember 中介層（middleware），這個中介層已經用來保護所有媒體相關的路由。如果該中介層的工作階段處理邏輯將來有任何改變（目前有一個關於訪客審查者工作階段的待辦票券 BL-2214），匯出下載的行為也會跟著變動。而這份 diff 裡面沒有任何機制能標示出這一點。
